# DB基礎メモ

---

## インデックスとは

本の**巻末索引**と同じ仕組み。特定のカラムの値とレコードの場所をあらかじめ整理しておくことで、検索を高速化する。

### インデックスなし（フルテーブルスキャン）
```
全レコードを1件ずつ順番に見ていく
→ レコードが増えるほど遅くなる
```

### インデックスあり
```
インデックスを見て対象レコードに直接アクセス
→ 10万件あっても一瞬
```

### トレードオフ
| | 検索速度 | 書き込み速度 | ストレージ |
|--|---------|------------|-----------|
| インデックスなし | 遅い | 速い | 少ない |
| インデックスあり | 速い | 少し遅い | 増える |

→ **「検索でよく使うカラム」にだけ貼る**のが正解

### 外部キーには必ずインデックスを貼る理由
- 外部キーは「関連するデータを取得する」ときに必ず使われる
- JOINのときにも使われる
- PostgreSQL（Supabase）は外部キーに**自動でインデックスを貼らない**ので手動で書く必要がある
  - ※ MySQLは自動で貼ってくれる

## 外部キー制約（REFERENCES）とは

```sql
owner_id UUID NOT NULL REFERENCES auth.users(id)
```

「このカラムの値は、指定したテーブルの指定したカラムに**必ず存在する値でなければならない**」という制約。

```
存在するIDを入れる  → OK ✅
存在しないIDを入れる → エラー ❌
```

→ 「どのユーザーのものかわからないデータ」が生まれるのを防ぐ

## ON DELETE の動作

親レコードを削除したときの子レコードの扱いを指定する。

### CASCADE（連鎖削除）
```sql
REFERENCES auth.users(id) ON DELETE CASCADE
```
親を削除すると子も**自動で削除される**。

```
オーナーのアカウントを削除
  ↓ CASCADE
店舗が削除
  ↓ CASCADE
キャスト・顧客・来店記録が削除
```

→ 孤立したデータが残らない。削除が楽。

### RESTRICT（削除制限）
```sql
REFERENCES auth.users(id) ON DELETE RESTRICT
```
子が存在する限り、親を**削除できない**。先に子を整理する必要がある。

```
店舗が存在するオーナーのアカウントを削除しようとする
  ↓
エラー 🚫「先に店舗を移譲・削除してください」
```

→ 大事なデータが誤って消えるのを防げる。

### 使い分け
| ケース | 推奨 |
|--------|------|
| 子は親なしに存在意義がない | CASCADE |
| 子は親が変わっても残すべき | RESTRICT |

**例：** オーナーが変わっても店舗・顧客データは残すべき → `stores.owner_id` は RESTRICT  
**例：** 店舗が消えたらキャストデータも不要 → `casts.store_id` は CASCADE
